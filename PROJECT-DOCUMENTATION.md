# 需求捕手 ReqCatcher
## 项目说明书

## 1. 项目概述

**项目名称**：需求捕手（ReqCatcher）  
**项目类型**：OpenClaw Skill  
**核心定位**：AI 产品架构师 · 模糊需求终结者  
**开源地址**：https://github.com/EvianEvans/requirement-clarifier

### 1.1 一句话介绍

需求捕手是一个 OpenClaw Skill，通过结构化对话帮助用户将模糊需求转化为精确可执行的需求规格文档，从根本上解决 AI 开发中的"需求返工"问题。

### 1.2 核心理念

> "想清楚，再动手。"

在 AI 编程时代，最大的时间黑洞不是代码编写，而是需求返工。需求捕手通过"渐进引导 + 参数化抽象 + 毒舌纠偏"三大机制，确保需求一次性对齐，避免反复修改。

---

## 2. 问题背景

### 2.1 行业痛点

**典型场景**：
```
产品经理说："帮我做一个简单的电商后台"
↓ 开发者开始编码
↓ 交付后
产品经理："啊，我没说要支持多语言"
           "这里应该能拖拽排序的"
           "用户权限应该分三层，不是两层"
↓ 返工 3 次，延期 2 周
```

**数据现状**（行业调研）：
- 软件项目平均返工率高达 40%
- 60% 的返工源于需求理解偏差
- 每次返工平均消耗 30% 的项目时间

### 2.2 根本原因

1. **需求模糊**：用户说"看着办"，实则隐含大量假设
2. **沟通断裂**：用户和开发者对"简单"的理解不同
3. **缺乏结构化**：没有系统化的需求澄清流程

### 2.3 现有方案不足

- **传统问卷**：一次性抛出 20 个问题，用户疲惫放弃
- **AI 直接生成**：ChatGPT 直接给方案，但不符合实际需求
- **人工产品经理**：成本高，无法规模化

## 3. 解决方案

### 3.1 系统架构

```
用户输入模糊需求
      ↓
┌─────────────────┐
│  Phase 1        │ 需求复杂度判定
│  - 简单需求     │ → 分轮次问卷（2-3问/轮）
│  - 复杂需求     │ → 对话引导模式
└────────┬────────┘
         ↓
┌─────────────────┐
│  Phase 1.5      │ 先做功课，再提问
│  - 研究案例     │ 
│  - 主动建议     │
└────────┬────────┘
         ↓
┌─────────────────┐
│  Phase 2        │ 毒舌纠偏
│  - 可行性检查   │
│  - 强制二选一   │
└────────┬────────┘
         ↓
┌─────────────────┐
│  Phase 3        │ 输出规格文档
│  - 业务侧       │（给人看）
│  - 技术侧       │（给机器看）
└─────────────────┘
```

### 3.2 三大核心能力

#### 3.2.1 渐进引导 Progressive Guidance

**解决**：一次性问卷过长，用户放弃

**机制**：
- 简单需求：分 2-3 轮提问，每轮 2-3 个核心问题
- 复杂需求：对话式"剥洋葱"，逐步深入
- 先做功课：主动提供建议，而非空白问卷

**案例**：
> 用户："我想要一个猫咪领养海报生成器"  
> 需求捕手："我研究了 8 种设计风格，推荐 3 种给你..."  
> （而非："你想要什么风格？什么配色？什么尺寸？"）

#### 3.2.2 参数化抽象 Parametric Abstraction

**解决**：硬编码导致一改全改

**机制**：
- 识别变量 vs 固定值
- 定义参数联动规则（A 变 → B、C 自动联动）
- 边界矩阵显性化（谁贴着谁，重叠如何处理）

**案例**（芯片封装建模）：
> 用户："设计一个芯片封装模型"  
> 需求捕手："如果 Die 尺寸变了，Bump 的行列数应该自动重算吗？"  
> 输出：参数化配置面板 + Z-Offset 联动机制

#### 3.2.3 毒舌纠偏 Brutal Honesty

**解决**：不切实际的需求导致无效劳动

**机制**：
- 可行性检查：不切实际需求直接打回
- 强制二选一：必须提供可行替代方案
- 零委婉：明确告知不可行及原因

**案例**：
> 用户："帮我做一个自动写小说的 AI，要超过 GPT-4，明天上线"  
> 需求捕手："这个需求不可行，原因有三：... 建议改为方案 A 或方案 B"

### 3.3 输出成果

**双轨输出**：

**[业务侧] 业务逻辑与用户体验（给人看）**
- 核心目标
- 交互流程
- 核心功能点
- 变量联动规则

**[技术侧] 技术参数与执行边界（给机器看）**
- 技术栈要求
- API 与数据结构
- 边界条件与异常处理
- 绝对禁区

### 4.3 工作原理

**触发条件**：
- 用户说"帮我做一个..."
- 检测到模糊需求（缺少关键参数）
- 跨领域复杂需求

**执行流程**：
1. 加载 SKILL.md 指令
2. 进入 Phase 1：判定复杂度
3. 根据分支执行对应流程
4. 输出需求规格文档
5. 写入 MEMORY.md 记录

## 5. 应用案例

### 5.1 案例一：电商后台需求澄清

**初始需求**："帮我做一个简单的电商后台管理系统"

**澄清过程**：
| 轮次 | 问题 | 用户回答 |
|:---|:---|:---|
| 1 | 后台类型？数据分析 vs 文档类 | 数据分析类 |
| 2 | 用户角色？ | 超级管理员、运营、客服 |
| 3 | 技术约束？ | 用现有 MySQL，不碰支付系统 |

**输出规格**：
```markdown
## 需求规格文档

### [业务侧]
- 目标：B2C 电商后台，支持商品/订单/用户管理
- 用户角色：超级管理员、运营人员、客服人员
- 核心功能：商品上架、订单处理、用户查询、数据报表

### [技术侧]
- 技术栈：React + Node.js + MySQL
- 边界条件：支持并发 100 用户，数据每日备份
- 绝对禁区：不涉及支付系统（使用第三方支付）
```

### 5.2 案例二：芯片封装几何建模（工业级复杂）

**初始需求**："设计一个芯片封装模型，能自动生成 Ansys 脚本"

**挑战**：
- 12 层基板结构
- 380 个 Bump 圆柱阵列
- 208 个 SolderBall 圆柱阵列
- 参数一改动全身

**解决方案**：

**1. 参数化逼问**：
> "如果 Die 尺寸变了，Bump 的行数和列数应该自动重算吗？"

**2. 物理空间逻辑确认**：
> "12 层结构中，哪些是紧密堆叠的？修改某层高度时，其他层如何联动？"

**3. 边界重叠矩阵**：

| 被减体 | 减去 | 原因 |
|:---|:---|:---|
| EMC | Die | Die 嵌入 EMC 内部 |
| EMC | Bump | Bump 穿过 EMC 范围 |
| Underfill | Bump | Bump 穿过 Underfill |

**4. 可视化渐进确认**：
> "实现思路：前三层固定不可拖，基板层支持排序。接受吗？"

**实际交付**：
- 3D 可视化：多层结构及 Bump/SolderBall 阵列
- XZ 截面图：层叠结构及参数化配置
- 参数化配置面板：直径/高度/间隔/基准面
- 布尔运算规则确认表

### 5.3 案例三：猫咪领养海报生成器

**初始需求**："我想要一个猫咪领养海报生成器"

**澄清过程**：

**先做功课（主动建议）**：
> "我研究了 8 种设计风格，为你推荐 3 种：
> - **温馨治愈系**：柔和渐变、圆润字体，适用幼猫
> - **现代极简系**：大量留白、ins 风，适用品种猫
> - **赛博朋克系**：霓虹光效、科技感，适用黑猫
> 你倾向哪种？"

**分轮确认**：
1. 风格方向 + 尺寸需求
2. 字段配置（必填/可选）+ 拖拽需求
3. 动态效果（GIF/短视频/微交互）

**实际交付**：
- 10 种艺术风格切换（新艺术运动/装饰艺术/包豪斯/波普艺术等）
- 实时换色（4 种配色方案）
- 照片上传与管理
- 可拖拽字段编辑
- 导出 PNG/JPG

## 6. 创新亮点

### 6.1 与传统方案对比

| 维度 | 传统问卷 | AI 直接生成 | 人工产品经理 | 需求捕手 |
|:---|:---|:---|:---|:---|
| **成本** | 低 | 极低 | 高 | 低 |
| **准确性** | 中 | 低 | 高 | 高 |
| **用户体验** | 差（长问卷） | 差（瞎猜） | 好 | 好（对话式）|
| **可扩展性** | 差 | 差 | 差 | 好（Skill 化）|

### 6.2 核心技术突破

**1. 先做功课，再提问**
- 传统：空白问卷让用户填空
- 需求捕手：研究后主动提供建议，让用户选择/调整

**2. 参数化抽象**
- 传统：硬编码，一改全改
- 需求捕手：变量识别 + 联动规则，自动计算派生值

**3. 毒舌纠偏**
- 传统：委婉暗示用户改需求
- 需求捕手：直接打回 + 强制二选一

### 6.3 实际效果

基于 3 个实战案例的反馈：
- 需求澄清时间：从平均 3 天缩短至 30 分钟
- 返工次数：从平均 3 次降至 0-1 次
- 用户满意度：从"经常扯皮"到"一次对齐"
